# R6 Supervised Milestone Autocycle Pilot

## Milestone name
`R6 Supervised Milestone Autocycle Pilot`

## Why this milestone exists now
R5 closed a real bounded foundation milestone, but the repo is still too governance-heavy and too thin on direct operator value.

The current substrate now includes governed planning and work-artifact foundations, QA and audit-pack foundations, baton persistence and continuity foundations, Git-backed milestone baseline capture, restore-gate validation foundations, bounded resume re-entry preparation, bounded proof replay and CI foundations, and repo-enforced proof packaging plus review structure. That is enough foundation to justify one integrated supervised pilot.

The next defensible step is still not UI expansion, not Standard or subproject runtime, not rollback productization, and not broad autonomy. The next bounded step is to prove one supervised, operator-controlled milestone autocycle from structured intake to operator decision inside this repository only.

## Objective
Prove one operator-controlled, fail-closed, evidence-first milestone autocycle pilot for `AIOffice_V2` only, from structured intake to final operator decision, without widening into broad autonomy, unattended automatic resume, rollback execution, UI productization, Standard runtime, or broader orchestration.

## Current open status
`R6 Supervised Milestone Autocycle Pilot` is open in repo truth through `R6-001` only.

This opening step defines the exact pilot boundary, stop conditions, initial contract foundation, and preserved non-claims. No later `R6` implementation task is complete yet.

## Exact pilot boundary
This milestone is bounded to:
- one repository: `AIOffice_V2`
- one active milestone cycle at a time
- one operator-approved milestone plan containing roughly 5 to 10 tasks
- one executor type only
- sequential task dispatch only
- one Git-backed baseline anchor per frozen milestone
- one QA observation path
- one PRO-style milestone summary path
- one final operator decision packet
- one replayable end-to-end pilot proof

## Exact stop conditions
The pilot must refuse or stop if:
- the milestone plan is not explicitly approved and frozen
- baseline capture fails
- repo state drifts outside allowed conditions
- executor evidence is incomplete
- QA evidence is incomplete or failed
- audit summary inputs are incomplete
- retry ceilings are exceeded
- proof selection or replay integrity is inconsistent

## Required milestone outputs
By the end of `R6`, the milestone must produce:
- a milestone-cycle contract set
- a milestone task-set proposal from one structured intake
- an explicit operator approval and freeze artifact
- a baseline link from the frozen milestone to a Git-backed milestone baseline
- a Codex dispatch contract and run ledger
- a governed execution evidence bundle assembler
- bounded QA observation and milestone aggregation
- a bounded PRO-style milestone review summary and operator decision packet
- one replayable proof for the full pilot
- repo truth and closeout surfaces that state the pilot honestly and preserve non-claims

## Preserved non-claims
R6 does not currently prove and must not casually widen into:
- broad autonomy or general orchestrator productization
- rollback execution
- unattended automatic resume
- UI or control-room productization
- Standard runtime or subproject runtime
- multi-repo behavior
- executor swarms
- broader orchestration beyond the exact supervised pilot

## In scope
- milestone-level intake to task-set proposal for one bounded pilot
- explicit operator approval and durable milestone freeze before dispatch
- binding the frozen milestone to one Git-backed baseline anchor
- governed Codex dispatch and one active run ledger
- governed execution evidence assembly from executor outputs and repo evidence
- bounded QA observation and milestone-level aggregation
- one bounded PRO-style summary and operator decision packet
- one replayable end-to-end pilot proof plus honest closeout packaging
- the immediate R6 preconditions needed to support those later steps honestly

## Explicitly out of scope
- any operator-visible or user-facing UI work
- unified workspace work
- multi-repo or fleet behavior
- more than one executor type
- parallel task dispatch or executor swarms
- automatic merge or promotion
- unattended automatic resume behavior
- rollback execution
- Standard or subproject runtime
- broad orchestration beyond the exact pilot boundary
- donor backlog import or donor milestone migration

## Dependencies and prerequisites
- `RST-009` through `RST-012` remain complete and externally accepted
- `R3-001` through `R3-008` remain complete in repo truth
- `R4-001` through `R4-011` remain complete in repo truth
- `R5-001` through `R5-007` remain complete and formally closed in repo truth

## Key risks
- spending R6 on governance hygiene instead of real supervised operator value
- allowing narration or summaries to outrun governed evidence
- permitting dispatch before freeze or before Git-backed baseline anchoring
- allowing shell-location-sensitive path handling to remain inside baton-related behavior
- widening the pilot into general orchestration instead of one exact supervised cycle
