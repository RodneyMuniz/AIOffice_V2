# AIOffice Vision

This file is the constitutional truth for AIOffice. If any other repo document conflicts with this file, this file wins.

## Product Thesis
AIOffice exists to make governed software production possible around untrusted models.

The product is not a model and not a chat wrapper. The product is a thin control harness that keeps operator intent, control-kernel authority, bounded execution, review, and durable truth separate.

V1 is admin-only and self-build first. It is allowed to be narrow as long as it is real, reviewable, and honest about what it proves.

## Doctrine
- Fail closed over fail open.
- Evidence outranks narration.
- Artifacts outrank summaries.
- Explicit approval outranks implied permission.
- One authoritative truth surface per concern.
- Current-proof honesty outranks product ambition.

## Authority Model
Authority order is:
1. operator
2. control kernel
3. reasoning or orchestration assistance
4. bounded executor

Executors may produce bounded work. They do not become the control plane. Chat may carry intent, but chat does not become silent authority.

## Product Shape
AIOffice is intended to be:
- API-first
- cost-governed
- execution-profile aware
- artifact-backed instead of narration-backed
- grounded in Git and persisted state as truth substrates

The harness is the product. Governance is not a sidecar.

## First Proof Boundary
The first acceptable proof boundary is:
- supervised operation through `architect`
- plus bounded `apply/promotion` control

The live collaboration workflow that must be proved stops at `architect`. Downstream implementation, design, QA, automation, or multi-lane orchestration are not required for the first proof.

The first real proof must include a reviewed path from approved artifacts into bounded mutation or promotion. A narrated claim of readiness is not enough.

## Current Non-Claims
The current repo does not claim:
- broad UI or control-room proof
- unattended operation
- broad later-lane workflow proof
- a Standard or subproject pipeline requirement in current V1
- legacy task, milestone, or kanban migration
- that product breadth beyond this boundary is already implemented

## Target-State Direction
Product ambition may extend beyond the first proof. AIOffice may later grow into a broader governed production surface with richer operator tooling and wider workflow coverage.

That ambition does not weaken present honesty. Anything beyond the first proof boundary is target-state only until it is explicitly proved.
